# Session Log: Render.com Deployment Setup - 2026-04-01 14:00 UTC+2

## Objective
Generate a `render.yaml` file for deploying ProjectsTracker to Render.com, adapting the Docker setup for a managed platform environment.

## Context
- Previous session: Created full Docker setup (Dockerfiles, docker-compose.yml)
- Challenge: Render.com assigns separate public URLs to each service
- Constraint: JWT cookies use `SameSite=Lax` — won't work cross-domain
- Solution: Preserve Nginx proxy architecture with environment variable substitution

## Key Problem Solved

**The Issue**: 
- Docker Compose: Frontend → `http://backend:5000` (docker hostname)
- Render: Each service gets its own URL (e.g., `https://projectstracker-api.onrender.com`)
- Nginx config is baked into Docker image at build time
- Frontend must proxy to backend's Render URL, which doesn't exist until deploy time

**The Solution**:
- Move nginx.conf → `/etc/nginx/templates/default.conf.template` (build time)
- Use `docker-entrypoint.sh` with `envsubst` to substitute `${BACKEND_URL}` (runtime)
- Use `fromService` references in render.yaml for Render to auto-set service URLs
- Maintain backward compatibility: Docker Compose gets `BACKEND_URL=http://backend:5000`

## Decisions Made

### 1. Preserve Nginx Proxy for Cookie Auth
- JWT cookies: `sameSite: 'lax'` (authRoutes.js:25)
- SameSite=Lax: NOT sent on cross-domain requests
- Rejected: Direct `VITE_API_URL` approach (would break auth)
- Accepted: Nginx proxy makes frontend/backend appear same-origin

### 2. Runtime Environment Substitution
- Can't template nginx.conf at build time (service URLs don't exist)
- Solution: `docker-entrypoint.sh` runs `envsubst` at container startup
- Only substitutes `BACKEND_URL`, preserves nginx variables like `$uri`

### 3. Service Auto-Discovery via fromService
- Render feature: `fromService` references auto-set env vars at deploy time
- Backend's `CORS_ORIGIN` ← frontend's `hostWithScheme`
- Frontend's `BACKEND_URL` ← backend's `hostWithScheme`
- Eliminates manual URL configuration

### 4. Starter Plan for Both Services
- Backend: Starter required for persistent disk
- Frontend: Starter for consistency (Free tier sleeps after 15 min)
- Cost: $7/month × 2 = $14/month

### 5. 1GB Persistent Disk for SQLite
- SQLite at `/app/data/database.sqlite`
- Disk survives container restarts
- 1GB sufficient for small-to-medium datasets

## Architecture Comparison

### Docker Compose (Local)
```
localhost → Nginx (port 80)
         → proxy /api/* → backend:5000 (docker hostname)
```

### Render.com (Production)
```
Browser → Nginx (projectstracker-frontend.onrender.com)
       → proxy /api/* → projectstracker-api.onrender.com (set via fromService)
```

Both use Nginx proxy, but Render version uses environment variables instead of hardcoded hostnames.

## Deployment Flow (for user)

```
1. git push render.yaml + modified files
2. Connect repo to Render Dashboard
3. Create Blueprint (Render detects render.yaml)
4. Deploy
   a. Render builds backend image
   b. Render builds frontend image (with VITE_API_URL build arg)
   c. Backend container starts → migrations run → /health endpoint up
   d. Frontend container starts → docker-entrypoint.sh runs:
      - Reads BACKEND_URL env var (set by fromService)
      - Runs: envsubst '${BACKEND_URL}' < template > config
      - Starts: Nginx with resolved backend URL
5. Both services live at xxx.onrender.com
6. Frontend Nginx proxies /api/* to backend
7. Auth cookies work (same-origin via proxy)
```

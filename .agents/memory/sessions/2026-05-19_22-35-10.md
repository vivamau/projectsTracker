## Inline tool-call rescue (Gemma/local-model fix)
Local models (Gemma via Ollama) sometimes emit tool calls as JSON text
inside `content` instead of using the native `tool_calls` field — the
JSON leaked to users and the SQL was never run.

- `extractInlineToolCall(text)`: parses `{name, parameters|arguments|input}`,
  handles fenced blocks. `stripSqlFromResponse` now also strips inline
  tool-call JSON.
- `executeInlineToolCall(inline, toolboxTools, db, trace)`: runs
  `execute_sql` via `runQueryDirectly` or invokes a matching toolbox
  tool; trace recorded so provenance still works.
- Wired into Ollama + OpenAI-compat loops as a third fallback after
  native `tool_calls` and raw-SELECT extraction. Claude/Gemini skipped
  (native protocols are reliable there).
- Integration test confirms inline JSON triggers real SQL execution and
  the visible answer is JSON-free. Full suite: 1306 tests pass.

## Per-message RAG/MCP toggle
- `chat()` accepts `useRag`/`useMcp` (defaults true). `useRag=false`
  skips `retrieveWithContext`. `useMcp=false` passes empty toolset to
  the provider API, skips raw-SQL extraction, skips inline tool-call
  rescue, and ignores native `tool_calls` (returns the text as-is).
  Applied to Ollama, OpenAI-compat (+GPT/NVIDIA/OpenRouter/llama.cpp),
  Claude, Gemini.
- `/api/agent/chat` accepts `useRag`/`useMcp` in body and forwards them
  (default true preserves existing behavior).
- Frontend: per-message dropdown next to Send — "RAG + MCP / RAG only /
  MCP only / None". State is per-message (re-pickable) and forwarded
  via `sendChatMessage(..., { useRag, useMcp })`.
